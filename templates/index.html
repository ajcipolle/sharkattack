Machine Learning Process

We built Random Forest and Decision Tree models using location and other features to predict the probability of a fatality of a shark attack. Output of our machine learning models fill our shark attack database.

For the machine learning aspect of this project, we chose classification models because, with this data set, we were predicting an outcome rather than a numerical value. To obtain more accurate predictions, we ran several different classification models and compared the results. We ran Logistical Regression, Random Forest Classification, and Decision Tree Classification models and each one displayed different results. We began by running the test data through each model.

Then, to verify accuracy we ran the training data through the models. We were genuinely surprised to see that the model was overfitting, finding that the training model had higher accuracy than the testing model. We also ran a feature importance test on the highest-scoring model, Random Forest Classification, and discovered that the injury column dominated all other features.

  • Features: injury, species of shark, time of day, activity, location, age of victim
  • Logistical Regression, Decision Tree, Random Forest, Cross-Validation Model, Feature Importance

Feature Importance

One thing we noticed was how much the injury feature played an extremely significant role in the results of our model looking at the graph of feature importance. We then ran the model without injury column as a feature. The results were drastically different. As you can see from the R^2 the accuracy went way down. The feature importance was a lot more even however.

Species was the highest but not at the margins that injury was when it was a feature. Another interesting note was that literally every prediction was N for surviving the shark attack. The best explanation for this is that by removing the injury column the model had no way of coming to a determination because none of the other columns had any information on whether the individual was harmed or not by the attack. With this in mind we chose to use the injury feature for our final analysis

Refining Our Model



Through more research, we discovered that the best solution was to run a cross-validation model to generate many models in order to then take the mean of each model group to see if the model stabilizes. Data Cleaning Process

The data cleaning process for this dataset was intense; there were many inconsistencies in the data. The largest offenders being: misspellings, extra spaces at the beginning or end of words, and detailed descriptions of the area instead of stating a location. Our plan when we were starting this project was to use a website and scrape the data. We planned on doing this by creating a loop that iterates through the area column and inputs each area before scraping the coordinates.

Unfortunately, this didn't work because the site we were using recognized and blocked the chrome driver after we scraped the data for three countries. After this setback, we decided to pivot because the coordinates we were pulling were not great matches due to the misspellings and the multiple returns we were receiving. We ended up using OpenCageGeocode API and used a replace_list to start the cleaning process. After that, we stripped the text of multiple variables to create a cleaner location for the API call.

To reduce the API calls, we created a text file that the areas were written to once they were cleaned. After we had a list of the cleaned areas from the text file, we created a dictionary, so we were able to store the response from the API call as a value with the area as a key. This step helped the data stay organized and allowed us to merge the response that we got back to the primary dataset as a test check. After we joined, we were able to find the locations that didn't get a match and manually enter as many as possible (about 6).

At this point, data cleaning and extraction were complete. We next dropped duplicates since some areas were mentioned much more often. We didn't need the duplicates in this situation because we were merging the geodata onto a different data set (output of machine learning, not the primary data). We left two options at the bottom of the Jupyter notebook for loading the data into the database. One is a CSV that you can manually load and one is a direct route to SQL.
